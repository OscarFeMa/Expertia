**Optimizing PostgreSQL queries: 12 indexing pitfalls and how we fixed them**

PostgreSQL is a powerful beast. But like anything powerful, it needs careful handling. At **Cubbit**, we use PostgreSQL as the **OLTP (Online Transaction Processing)** system to manage the metadata for users’ files — things like file names, paths, sizes, ownership, and timestamps.

This workload involves a high volume of small, concurrent transactions, and performance is key. We rely heavily on indexes to keep things fast and responsive. Along the way, we’ve uncovered a number of subtle (and sometimes surprising) indexing behaviors that are worth sharing, whether due to planner decisions, index structure, or data characteristics.

This post is a behind-the-scenes look at **12 real-world scenarios** where Postgres didn’t use an index (or used it inefficiently) and how we applied **PostgreSQL index best practices** and **query optimization techniques** to fix them.

## 🔍 PostgreSQL query plans: a quick primer

Before diving into the problems, here is a quick word on **PostgreSQL query plans**: This is how you understand what the planner is actually doing.

You can inspect a query plan with:

`EXPLAIN SELECT * FROM my_table WHERE …;`

This will **not execute** the query. It simply shows how PostgreSQL *intends* to run it. If you add ANALYZE, like this:

`EXPLAIN ANALYZE SELECT * FROM my_table WHERE …;`

…it will **actually execute** the query and return detailed execution statistics (timings, row counts, etc.).

⚠️ Be careful: if your query is an `INSERT`

, `UPDATE`

, or `DELETE`

, then the `EXPLAIN ANALYZE`

will **run the operation for real**, modifying data just as if you had executed the statement normally. Always double-check the query and run it only in safe environments.

📘[ PostgreSQL query plans: EXPLAIN documentation](https://www.postgresql.org/docs/current/using-explain.html)

### 🧪 A note on test environments

When checking PostgreSQL query plans, do it on **well-populated tables**, ideally with realistic data distributions. On small or empty tables, PostgreSQL’s planner may choose different strategies (like sequential scans) that don’t reflect production behavior. Plans only make sense when the underlying statistics are representative.

Start with plain `EXPLAIN`

to understand query structure, and use `ANALYZE`

only when it’s safe to execute the statement.

## 🧵 Our approach to PostgreSQL query optimization techniques

Everything in this article is rooted in practice and reflects how we fine-tuned our own queries at scale. These PostgreSQL query optimization techniques aren’t theoretical. They come from real-world problems.

## 1. PostgreSQL partial index was not used because the query wasn’t explicit enough

We had a `files`

table where every file with `size = 0`

always had a `NULL file_hash`

.

So we created a **PostgreSQL partial index** (*index with a WHERE condition*):

`CREATE INDEX ON files (size)`

WHERE size = 0 AND file_hash IS NULL;

But Postgres didn’t use it when we ran:

`SELECT * FROM files WHERE size = 0;`

The query plan showed a **sequential scan**, which was a red flag.

Why? **Partial indexes only work when the query WHERE clause matches the index condition**. Even though we *knew* `file_hash`

would be `NULL`

, Postgres didn’t.

### 🔧 The fix

Just be explicit.

`SELECT * FROM files WHERE size = 0 AND file_hash IS NULL;`

This is a good example of why **PostgreSQL index best practices** emphasize explicitly matching WHERE conditions (especially when dealing with partial indexes).

📘[PostgreSQL Docs: Partial indexes](https://www.postgresql.org/docs/current/indexes-partial.html)

## 2. PostgreSQL composite index on multiple columns with wrong order

We had this query:

`SELECT * FROM logs`

WHERE user_id = 42 AND created_at > now() - interval '1 day';

And this composite index (*a multicolumn index containing more columns*):

`CREATE INDEX ON logs (created_at, user_id);`

But it wasn’t used.

The mistake? In a **PostgreSQL composite index** on multiple columns, order matters. PostgreSQL optimizes best when the **index column order matches the query’s filtering pattern**.

In general:

- Place columns with
**equality conditions (=)**first - Follow with
**range conditions**(`>`

,`<`

,`BETWEEN`

, etc.) - Then put columns used for
**sorting**(`ORDER BY`

) or less selective filters

Why? Because PostgreSQL can only efficiently use the index for the leading columns in the order defined, once it hits a non-equality condition, it may stop using later columns for filtering.

### 🔧 The fix

So we switched to:

`CREATE INDEX ON logs (user_id, created_at); - they swapped order`

Now, Postgres could do an efficient index scan.

We added this case to our internal list of **PostgreSQL query optimization techniques**, as misordered composite indexes are a frequent source of missed performance gains.

📘[PostgreSQL Docs: Multicolumn indexes](https://www.postgresql.org/docs/current/indexes-multicolumn.html)

## 3. Index not selective enough: We added a second column

We had a query like this:

`SELECT * FROM downloads WHERE user_id = 42 AND download_status = 'complete';`

Originally, we had an index on just `user_id`

:

`CREATE INDEX ON downloads (user_id);`

Postgres used the index, but the query plan showed it by applying a **Filter** on `download_status`

. That meant:

- It used the index to find all rows with
`user_id = 42`

- But then it checked
`download_status = ‘complete'`

**after**fetching each row

This is inefficient if the user has many downloads, especially if most don’t match the `download_status`

.

Why does this happen? In PostgreSQL query plans:

**Index Cond**shows conditions that the index can use directly to find matching rows**Filter**shows conditions that are checked*after*the rows are retrieved

So a plan like this:

`Index Scan using downloads_user_id_idx on downloads`

Index Cond: (user_id = 42)

Filter: (download_status = 'complete')

This means the index only helped with `user_id`

, not with `download_status`

.

### 🔧 The fix

Create a **PostgreSQL composite index** to cover both columns:

`CREATE INDEX ON downloads (user_id, download_status);`

Now the query plan showed **both conditions in the Index Cond**, and the Filter step disappeared. This meant Postgres could retrieve only the matching rows directly from the index — faster and more efficient.

💡 Always check the query plan: if you see a Filter, you’re probably reading more rows than necessary. Ideally, all filtering should happen inside the Index Cond.

## 4. LIKE query didn’t use the index due to PostgreSQL collation settings

We had a query like this:

`SELECT * FROM files WHERE path LIKE '/media/photos/2023/%';`

In PostgreSQL, prefix-style LIKE queries similar to this are internally optimized to range comparisons:

`WHERE path >= '/media/photos/2023/' AND path < '/media/photos/2024/'`

A standard B-tree index *should* be usable:

`CREATE INDEX ON files (path);`

However, Postgres **ignored the index** and did a sequential scan instead.

Why? The problem was with the **PostgreSQL collation** on the `path`

column: specifically, it was set to `en_US.UTF-8`

. This collation uses locale-aware rules for comparing strings, which can lead to **non-binary-consistent ordering**. PostgreSQL cannot safely use B-tree indexes for range conditions (`>=`

, `<`

, or `LIKE ‘prefix%’`

) unless it can guarantee that string comparisons are consistent with the index order.

💡 Indexes only work for range comparisons on strings when the collation ensures predictable ordering.

### 🔧 The fix

Instead of just altering the index, we changed the **default PostgreSQL collation** on the database to `"C”`

, which uses simple **binary (byte-by-byte) ordering**. This guarantees that `’a’ < ‘b' < ‘c'`

behaves exactly as expected — no locale rules involved.

After switching to `"C”`

, PostgreSQL could reliably use the index for these queries, and performance improved significantly.

This collation tweak is often overlooked in guides on **optimizing PostgreSQL queries**, but it can make or break performance for LIKE queries with B-tree indexes.

To check the current collation of a column:

`SELECT column_name, collation_name`

FROM information_schema.columns

WHERE table_name = 'files';

📘[ PostgreSQL Docs: Collation support](https://www.postgresql.org/docs/current/collation.html)

## 5. Force Postgres to use an index with the correct ordering

Imagine we have a query like this:

`SELECT * FROM items`

WHERE category = 'books'

ORDER BY published_at DESC

LIMIT 10;

We initially had an index like this:

`CREATE INDEX ON items (category, published_at);`

**PostgreSQL** **was not using the index**. Instead, it was scanning more rows and doing an explicit sort step.

Why? The ordering in the query (`ORDER BY published_at DESC`

) **didn’t match** the index’s default sort order (which is ascending). Indexes are only useful for `ORDER BY`

if the ordering of columns in the index matches the query exactly, both in **column order** and **direction** (ASC/DESC).

### 🔧 The fix

We recreated the index in descending order:

`CREATE INDEX ON items (category, published_at DESC);`

Now, Postgres could scan the index directly in the right order and avoid sorting, making the query much faster.

## 6. PostgreSQL BRIN index on timestamps and time-series data

For a large metrics table:

`CREATE TABLE metrics (`

recorded_at timestamptz,

value float

);

We used:

`CREATE INDEX ON metrics USING BRIN (recorded_at);`

**PostegreSQL BRIN index (Block Range INdexes)** stores summaries (like min/max values) for each range of physical blocks on disk, rather than individual row pointers like B-tree indexes. This makes them **extremely space-efficient**, especially on huge tables.

But there’s a catch: PostgreSQL BRIN index only works well when the indexed column **physically orders** the data on disk. That’s common in **append-only time-series tables**, where new rows are inserted in chronological order.

### 🧪 Example: a good PostgreSQL BRIN use case

Imagine you’re recording one million sensor readings per day, each with a timestamp. New rows are appended in order. A query like:

`SELECT * FROM metrics WHERE recorded_at >= now() - interval '1 day';`

…can be satisfied by scanning just the relevant BRIN blocks near the end of the table, rather than the entire thing. That’s a huge performance win.

### ⚠️ When PostgreSQL BRIN index won’t help

If your `recorded_at`

values are updated often, inserted out of order, or the table isn’t massive, BRIN’s summaries become unreliable, and you may scan more blocks than you’d like. In such cases, a B-tree index might perform better despite being larger.

Choosing between BRIN and B-tree is one of the more strategic **PostgreSQL index best practices**, especially for time-series or append-only data.

📘[ PostgreSQL Docs: BRIN indexes](https://www.postgresql.org/docs/current/brin-intro.html)

## 7. Covering index to avoid table lookup

We had a query like this:

`SELECT name, updated_at FROM users WHERE email = 'john@example.com';`

Postgres was using this index:

`CREATE INDEX users_email_idx ON users (email);`

The query plan showed an **index scan** on `users_email_idx`

but it still had to **fetch rows from the table** to retrieve `name`

and `updated_at`

since those columns weren’t in the index.

That meant more disk I/O, even though we were using the index.

### 🔧 The fix

We created a **covering index** to include the needed columns:

`CREATE INDEX ON users (email) INCLUDE (name, updated_at);`

This allowed Postgres to perform an **index-only scan,** which is faster, because all required data was available from the index itself and it didn’t have to touch the main table.

📘[ PostgreSQL Docs: INCLUDE in indexes](https://www.postgresql.org/docs/current/sql-createindex.html#SQL-CREATEINDEX-INCLUDE)

## 8. PostgreSQL DELETE record was slow due to a missing index on the foreign key

We had this slow query:

`DELETE FROM region WHERE id = $1;`

Initially, the PostgreSQL query plan looked fine and showed an index scan. However, some metrics pointed to high CPU usage related to this query. We’ve then run again the query plan, this time with the ANALYZE, and something else came up:

`"Triggers": [`

{

"Trigger Name": "RI_ConstraintTrigger_a_21323",

"Constraint Name": "file_region_id_fkey",

"Relation": "region",

"Time": 10244.169,

"Calls": 126

}

],

...

The constraint `file_region_id`

is a foreign key we use to connect the `file`

and `region`

tables, and the time spent checking this constraint was around 10 seconds, which is too high for our scenario. But what’s this constraint about? On delete, Postgres must validate:

`SELECT 1 FROM file WHERE region_id = $1;`

There was no index on file.region_id, so it triggered a sequential scan for every DELETE, causing all that time spent checking the constraint.

### 🔧 The fix

`CREATE INDEX ON file (region_id);`

✅ Lesson: Postgres will silently perform costly checks to enforce referential integrity. Index your foreign keys.

## 9. Avoiding overindexing to preserve HOT updates

We had a performance issue where frequent updates on a table were unexpectedly expensive. Digging into it, we realized our updates weren’t benefiting from **HOT updates** (Heap-Only Tuples).

### What are HOT updates?

A **HOT update** is an optimization in PostgreSQL that allows the database to update a row **without modifying indexes**, as long as the indexed columns are unchanged. This means:

- Less I/O
- Less bloat in indexes
- No need to update visibility maps
- Faster updates

But this only works when:

- The updated columns are
**not**part of any index. - There is room on the same page for the new row version.

If any updated columns are indexed, PostgreSQL must perform a regular update and **touch the index**, negating the benefits of HOT.

### How to check the HOT update ratio

`SELECT n_tup_upd, n_tup_hot_upd,`

round(100.0 * n_tup_hot_upd / NULLIF(n_tup_upd, 0), 2) AS hot_update_ratio

FROM pg_stat_all_tables

WHERE relname = 'your_table';

### Our case: overindexed on a frequently updated column

We had an index like:

`CREATE INDEX ON documents (modified_at);`

But `modified_at`

was being updated often. Every update invalidated this index, **preventing HOT updates**.

We realized we didn’t actually need this index — it was only used for an occasional admin query that could tolerate a slower scan.

### 🔧 The fix

We dropped the index:

`DROP INDEX documents_modified_at_idx;`

Afterward, the HOT update ratio improved significantly, and frequent updates became much cheaper.

In this case, minimizing overindexing directly contributed to **optimizing PostgreSQL queries** during frequent updates, while preserving HOT optimization.

✅ Lesson: Be strategic about which columns you index — if they’re frequently updated and not part of any critical query filters, indexing them could hurt performance more than it helps.

📘[ PostgreSQL Docs: HOT updates](https://www.postgresql.org/docs/current/storage-hot.html)

## 10. Most queries filtered on one column → PostgreSQL table partitioning

We noticed that almost every query on our users table filtered by region_id.

This was a strong candidate for **PostgreSQL table** **partitioning**.

### 🔧 The fix

We changed the table to:

`CREATE TABLE users (`

id uuid,

name text,

region_id int

) PARTITION BY HASH (region_id);

Then, created child tables per partition. This allowed PostgreSQL to **prune partitions** and skip scanning irrelevant data, improving performance significantly.

**11. PostgreSQL stopped using the index after batch DELETE**

We had a query like this that used to be fast:

`SELECT * FROM some_table WHERE user_id = 123;`

Postgres used an index on user_id, and the plan showed an efficient index scan. But one day, with no code changes, the query **suddenly switched to a sequential scan,** and performance tanked.

### 🕵️ What changed?

It turned out that **a large batch DELETE** had recently run on the table:

`DELETE FROM some_table WHERE created_at < now() - interval '90 days';`

This removed **millions of rows**, but Postgres didn’t immediately update its internal statistics. As a result, the query planner had **stale row count estimates,** so it guessed that many rows matched the condition and decided a sequential scan would be faster.

We confirmed this by checking the last time ANALYZE had run:

`SELECT relname, last_autoanalyze`

FROM pg_stat_all_tables

WHERE relname = 'some_table';

The timestamp showed that **autovacuum hadn’t yet analyzed** the table after the big DELETE.

### 🔧 The fix

We manually refreshed the statistics:

`ANALYZE some_table;`

Immediately, the planner returned to using the index, and performance recovered.

This reinforced the importance of autovacuum timing as part of our broader **PostgreSQL query optimization techniques** checklist.

## 12. PostgreSQL index-only scan queries were slow due to index bloat

One query was using an **index-only scan**, yet it was still slow.

We checked the plan, and it correctly avoided table access. But the performance was poor.

Our suspicion: **index bloat**. This happens when deleted rows leave empty slots in the index that accumulate over time.

To check:

`SELECT pg_size_pretty(pg_relation_size('your_index'));`

You can also use the `pgstattuple`

extension for deeper insights.

### 🔧 The fix

`VACUUM (INDEX_CLEANUP ON) your_table;`
